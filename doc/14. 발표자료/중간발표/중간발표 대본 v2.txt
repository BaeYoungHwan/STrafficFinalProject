======================================================================
1. 팀 & 문제 정의 (Hook)
======================================================================

안녕하십니까 KDT 선도기업과정 종로 1팀 최종 팀프로젝트 중간발표를 맡은 팀장 배영환입니다.

발표는 팀 소개와 문제 정의를 시작으로, 프로젝트 개요, 시스템 아키텍처, 핵심 기능, DB 설계,
그리고 개발 과정에서의 이슈와 향후 계획 순으로 진행하겠습니다.

---

먼저 저희가 해결하고자 했던 문제를 말씀드리겠습니다.

현재 교통 시스템을 보면 CCTV 감시, 단속, 교통 흐름 관리, 설비 모니터링이
각각 따로 운영되고 있는 경우가 많습니다.

이로 인해 실시간 대응이나 데이터 연계 측면에서 개선의 여지가 있다고 판단했습니다.

이 문제를 해결하기 위해 저희는 3인 팀으로 프로젝트를 진행했습니다.

---

저는 PM으로서 전체 설계와 대시보드, CCTV 단속,
그리고 AI 서버 및 OCR 기반 번호판 인식을 담당했습니다.

김소연 팀원은 UI/UX와 DB 설계 및 테스트를,
하재영 팀원은 실시간 데이터 수집과 인증 시스템, 예지보전 AI를 담당했습니다.

---

저희는 6개월 동안 학습한 웹 풀스택과 AI 기술을 바탕으로
이 문제를 하나의 방향으로 해결하고자 했습니다.

즉, 개별적으로 동작하던 시스템을 하나의 흐름으로 통합하는 것이 목표였습니다.

그 결과 구현한 시스템이 바로 AIpass, 통합 교통 관제 시스템입니다.

3. 전체 시스템 아키텍처
======================================================================

"이제 AIpass의 전체 시스템 구조를 설명드리겠습니다."

"저희 시스템은 기능별 역할을 분리한
세 개의 독립적인 서버 구조로 설계되어 있습니다."

"먼저, 프론트엔드는 사용자가 직접 보는 화면을 담당하며
Vue 3 기반으로 구현했습니다."

"백엔드는 Spring Boot 기반으로
비즈니스 로직 처리와 DB 연동을 담당합니다."

"마지막으로 AI 서버는 FastAPI 기반으로,
객체 감지와 속도 계산, 번호판 인식 등
모든 AI 처리를 전담합니다."

"즉, 화면, 로직, AI 처리를 완전히 분리한 구조입니다."

---

"이제 실제 데이터가 어떻게 흐르는지 말씀드리겠습니다."

"사용자가 브라우저에서 시스템에 접속하면,
프론트엔드는 백엔드 서버에 API 요청을 보냅니다."

"한편 AI 서버는
CCTV 영상을 실시간으로 받아 차량을 감지하고 속도를 계산합니다."

"그리고 위반이 발생하면
해당 정보를 Webhook 방식으로
백엔드 서버에 자동으로 전달합니다."

"백엔드는 이 데이터를 DB에 저장하고,
프론트엔드는 저장된 정보를 다시 조회하여
사용자 화면에 표시하게 됩니다."

"이 과정을 통해
'감지 → 분석 → 저장 → 시각화'가
하나의 흐름으로 연결됩니다."

---

"이 구조에서 중요한 점은
각 서버가 서로 독립적으로 동작한다는 것입니다."

"특히 AI 연산은 부하가 크기 때문에
별도의 서버로 분리함으로써
전체 시스템 성능과 안정성을 확보했습니다."

"즉, 특정 기능에 문제가 발생하더라도
전체 서비스에는 영향을 최소화할 수 있도록 설계했습니다."

---

"오늘 시연은 이 구조를 실제로 확인하실 수 있도록
두 대의 컴퓨터로 진행됩니다."

"AI 서버는 제 환경에서,
프론트엔드와 백엔드는 시연 환경에서 각각 실행됩니다."

"이는 물리적으로 분리된 환경에서도
시스템이 정상적으로 연동된다는 것을
직접 보여드리기 위함입니다."

"그럼 지금부터
이 구조 위에서 동작하는 핵심 기능들을
순서대로 시연해 드리겠습니다."

4. 핵심 기능

----------------------------------------------------------------------
4-1. 인증 시스템
----------------------------------------------------------------------

"핵심 기능 첫 번째, 인증 시스템입니다."

"이 시스템은 일반 사용자가 아닌
관제 담당자를 위한 시스템이기 때문에
접근 제어가 매우 중요합니다."

"따라서 로그인 기반으로
인증된 사용자만 시스템에 접근할 수 있도록 구성했습니다."

"회원가입 시에는 실제 이메일로 6자리 인증코드를 발송하고,
5분 이내 입력을 완료해야 가입이 완료되는 구조로 구현했습니다."

"로그인, 아이디 찾기, 비밀번호 재설정까지
기본적인 인증 흐름을 모두 구현했습니다."

"인증은 HttpSession 기반으로 구현하였으며,
서버에서 세션을 관리하고"

"Interceptor를 통해 인증되지 않은 요청은
자동으로 로그인 페이지로 리디렉션됩니다."

"이를 통해
인가되지 않은 접근을 차단하고,
시스템의 안정성을 확보했습니다."

---

(로그인 화면 시연)

"지금 로그인 화면을 직접 보여드리겠습니다."

"ID와 비밀번호를 입력하고 로그인 버튼을 누르면,
인증된 관제 담당자로서 대시보드로 바로 이동합니다."

"반대로 로그인 없이 대시보드 URL에 직접 접근하면
자동으로 로그인 페이지로 리디렉션됩니다."

"이처럼 URL만 알고 있으면 접근이 차단되는 구조입니다."

----------------------------------------------------------------------
4-2. 대시보드
----------------------------------------------------------------------

"다음은 대시보드 화면입니다."

"대시보드는 관제 담당자가 가장 먼저 확인하는 화면으로,
현재 교차로 상황을 한눈에 파악할 수 있도록 설계했습니다."

"관제 시스템에서 중요한 것은 데이터 양이 아니라
판단 속도라고 생각했습니다."

"화면 상단에는 카카오맵 기반의 지도가 표시되며,
강화도 교차로 위치를 마커로 확인할 수 있습니다."

"각 CCTV 마커를 클릭하면
해당 교차로의 실시간 영상을 모달로 바로 확인할 수 있습니다."

(인화삼거리 마커 클릭 → 야간 라이브 팝업 확인)

"예를 들어, 인화삼거리 마커를 클릭하면
지금 이 순간의 교차로 실시간 영상이 바로 열립니다."

"별도의 화면 이동 없이
대시보드 내에서 즉시 상황 확인이 가능합니다."

(팝업 닫기 → 지도 복귀)

"이를 통해 관제 과정에서의 불필요한 이동을 줄이고,
운영 효율성을 높이고자 했습니다."

"지도 하단에는 상행, 하행, 전체 방향 필터를 제공하여
특정 구간의 혼잡도만 선택적으로 확인할 수 있습니다."

(하행↓ 버튼 클릭 → 하행 구간 혼잡도만 표시 → 전체 버튼 복귀)

"이를 통해 필요한 정보에 빠르게 집중할 수 있습니다."

"지도 아래에는 주요 정보를 요약한 위젯들이 배치되어 있습니다."

(하단 카드 영역으로 스크롤)

"CCTV 12대 전체 정상 운영 중,
오늘 단속 건수 210건, 현재 기온 13°C 강화군 기준 등
관제에 필요한 정보를 한눈에 파악할 수 있습니다."

"모든 위젯은 1분 단위로 자동 갱신되며,
화면 우측 상단에서 갱신 주기를 관제자가 직접 설정할 수 있습니다."

"결과적으로 대시보드는
여러 정보를 개별적으로 확인하는 것이 아니라,
한 화면에서 빠르게 상황을 판단할 수 있도록 설계했습니다."

----------------------------------------------------------------------
4-3. CCTV 모니터링
----------------------------------------------------------------------

"다음은 CCTV 모니터링 화면입니다."

"교통 관제에서 가장 기본이 되는 것은
현재 상황을 직접 눈으로 확인하는 것입니다."

"여러 교차로를 어떻게 효율적으로 동시에 모니터링할 것인가가
핵심 문제였습니다."

---

"저희가 선택한 방식은 그리드 구조입니다."

"강화도 ITS CCTV는 총 12개로 구성되어 있으며,
이를 한 화면에 모두 표시하도록 설계했습니다."

"덕분에 관제 담당자는 별도의 전환 없이
강화도 전 구간의 상황을 한 화면에서 동시에 파악할 수 있습니다."

---

"그리드 구조를 선택한 데는 또 다른 이유가 있습니다."

"실제 관제 상황에서는 이상 징후가
어느 교차로에서 발생할지 예측할 수 없습니다."

"모든 채널을 동시에 보면서
이상이 감지된 화면을 즉시 클릭해서 확대할 수 있어야
빠른 판단이 가능합니다."

"즉, 특정 위치를 찾는 방식이 아니라
전체를 동시에 감시하는 구조를 선택했습니다."

"저희는 이 점을 고려하여
개별 채널 클릭 시 해당 영상이 즉시 확대되도록 구현했습니다."

---

(CCTV 화면 전환 — 12채널 그리드)

"실제 화면을 보시면,"

"강화도 국도 48호선의 12개 교차로 영상이
동시에 스트리밍되고 있습니다."

"강화대교, 대산교차로, 부근교차로 등
각 채널 하단에 교차로명이 명확히 표시됩니다."

"모든 채널을 동시에 보면서
이상이 감지되는 위치를 즉시 식별할 수 있습니다."

(강화대교 채널 클릭 → 전체 화면 확대)

"강화대교 채널을 클릭하면
야간 라이브 영상이 전체 화면으로 확대됩니다."

"영상 하단의 강화 ↓ 서울 ↑ 방향 표시를 통해
상행·하행 차량 흐름도 즉시 파악할 수 있습니다."

(우측 상단 닫기 버튼 → 12채널 그리드 복귀)

"다시 전체 그리드로 돌아오는 것도 클릭 한 번입니다."

---

"또한 12개 채널을 동시에 스트리밍하는 것은
서버 입장에서 상당한 부하입니다."

"이를 고려하여 MJPEG 방식으로 스트리밍을 구성했으며,
각 채널의 해상도와 FPS를 제어하여
안정적인 동시 재생이 가능하도록 최적화했습니다."

"향후 채널 수가 늘어나더라도
이 구조 위에서 자연스럽게 확장 가능합니다."

---

"이 구조를 기반으로 단순 모니터링을 넘어
자동 분석 시스템으로 확장했습니다."

"DB에 설정된 특정 CCTV 영상이
FastAPI 기반 AI 서버로 자동 전달되고,"

"AI가 해당 영상을 실시간으로 분석하여
과속 감지와 번호판 인식을 수행합니다."

"즉, 관제 담당자가 영상을 단순히 '보는' 것이 아니라,
AI가 먼저 이상을 감지하고
그 결과를 화면에 즉시 반영하는 구조입니다."

"즉, CCTV 모니터링과 AI 분석이 분리된 기능이 아니라,
하나의 실시간 자동 처리 파이프라인으로 동작하도록 설계했습니다."

----------------------------------------------------------------------
4-4. 과속·위반 단속
----------------------------------------------------------------------

"이번이 저희 프로젝트의 핵심 기능입니다."

"실제 단속이 사람이 개입하지 않고
어떻게 자동으로 이루어지는지 보여드리겠습니다."

"전체 흐름은 세 단계입니다. 감지, 인식, 저장입니다."

"각 단계마다 저희 팀이 어떤 기술적 고민을 했는지
함께 설명드리겠습니다."

---

(YOLO 탐지 영상 재생 — 강화대교 차량 추적 장면)

"먼저 감지 단계입니다."

"저희는 객체 감지 모델로 YOLO를 선택했습니다."

"객체 감지 모델은 정확도 중심과 속도 중심으로 나뉘는데,
관제 시스템에서는 1초에도 여러 대의 차량이 지나가기 때문에
실시간 처리 속도가 훨씬 더 중요합니다."

"YOLO는 단일 신경망으로 감지와 분류를 동시에 처리하여
실시간 영상 분석에 최적화된 모델입니다."

"그 중에서도 경량 모델인 nano 버전을 사용했는데,
GPU 자원이 제한된 환경에서도
안정적으로 실시간 처리가 가능하기 때문에 선택했습니다."

---

"그런데 차량을 감지하는 것만으로는 부족합니다."

"화면에 차량이 여러 대 있을 때,
같은 차량을 프레임마다 동일한 개체로 인식해야
속도 계산이 가능합니다."

"이를 위해 ByteTrack 트래킹 알고리즘을 적용했습니다."

"여러 트래킹 알고리즘 중 ByteTrack을 선택한 이유는
낮은 confidence의 객체까지 활용하여 추적을 유지하기 때문입니다."

"CCTV 환경에서는 차량이 부분적으로 가려지거나
탐지 신뢰도가 순간적으로 떨어지는 경우가 많습니다."

"ByteTrack은 이러한 상황에서도
안정적으로 추적을 이어갈 수 있어 선택했습니다."

"또한 승용차, 오토바이, 버스, 트럭 네 가지 차종만
필터링하여 불필요한 감지 오류를 줄였습니다."

---

"이 부분이 속도 계산에서 가장 중요한 핵심입니다."

"화면 속 차량의 픽셀 이동을
어떻게 실제 도로 위 속도로 변환하는지입니다."

"카메라는 2D 평면으로 세상을 바라봅니다."

"화면에서 차량이 100픽셀 움직였다고 해서
실제로 몇 미터를 이동한 것인지는 바로 알 수 없습니다."

"카메라 각도, 거리, 렌즈 왜곡에 따라
같은 픽셀 이동이 완전히 다른 실제 거리를 의미하기 때문입니다."

"이 문제를 해결하기 위해 호모그래피 변환을 적용했습니다."

"호모그래피란, 카메라 영상의 2D 좌표를
실제 세계의 미터 단위 좌표로 변환하는 수학적 행렬입니다."

"호모그래피 변환은 최소 4개의 대응점이 필요하기 때문에,
강화도 교차로 도로 위 기준점 4개를 선정하여
픽셀 좌표와 실세계 미터 좌표를 매핑했습니다."

"구체적으로는 4차선 도로의 55미터 구간을 기준으로
행렬을 계산했습니다."

"이를 통해 차량의 픽셀 이동을
실제 미터 단위 이동 거리로 변환할 수 있습니다."

---

"그런데 프레임마다 계산된 순간 속도가
튀는 경우가 발생합니다."

"한 프레임에 72km/h,
다음 프레임에 갑자기 95km/h,
그 다음 프레임에 다시 68km/h처럼
불안정한 값이 나올 수 있습니다."

"카메라 노이즈, 차량 흔들림, 탐지 좌표 오차 등
물리적 한계에서 발생하는 문제입니다."

"이를 해결하기 위해 EMA,
지수이동평균 스무딩을 적용했습니다."

"EMA는 최근 값에 더 높은 가중치를 주어
급격한 속도 변동을 완화하는 방식입니다."

"알파값은 여러 실험을 통해 0.3으로 설정했으며,
노이즈 제거와 반응성 사이의 균형을 맞추는 값으로 선택했습니다."

"또한 물리적으로 불가능한 100km/h 이상의 값은
자동으로 제거되도록 상한선을 설정했습니다."

---

"위반 판정은 단순히 한 프레임에서 속도가 높다고
바로 단속하지 않습니다."

"70km/h를 초과한 프레임이 연속으로 5회 이상
지속될 때만 위반으로 확정합니다."

"이는 순간적인 오탐을 방지하기 위한 설계이며,
동일 차량은 한 번만 처리되도록 중복 전송도 차단했습니다."

---

(단속 내역 화면 전환 — 위반 차량 테이블)

"위반이 확정되면, 해당 차량의 번호판 이미지를
PaddleOCR이 인식합니다."

"OCR은 인식 실패 시 전처리 방식을 바꿔가며
최대 4단계까지 재시도하는 폴백 구조로 설계했습니다."

"단일 시도로 끝내지 않고 다양한 전처리를
단계적으로 적용한 이유는,
저해상도 CCTV 환경에서 인식 성공률을 높이기 위해서입니다."

"인식된 번호판은 신뢰도 0.85 이상인 경우 자동 승인으로,
그 이하는 수동 검토로 분류됩니다."

"위반 정보는 Webhook을 통해 백엔드로 즉시 전송되며,
전송 실패 시에도 DLQ에 자동 저장되어
5분 주기로 재전송이 시도됩니다."

"네트워크 오류가 발생하더라도
단속 데이터가 유실되지 않도록 설계했습니다."

---

"단속 내역은 실시간으로 테이블에 누적됩니다."

(1번 항목 — 131두8087 클릭 → 우측 상세 패널 열림)

"항목을 클릭하면 오른쪽에 상세 정보 패널이 열립니다."

"상단에는 차량 전체 이미지,
그 아래에는 번호판 크롭 이미지가 나란히 표시됩니다."

"이 항목을 보시면,
차량번호 131두8087,
측정 속도 80.4km/h,
위치 강화대교,
처리 상태 승인으로 확정되어 있습니다."

"반면 UNRECOGNIZED로 표시된 항목들은
OCR 인식에 실패한 케이스로,
관제자가 이미지를 직접 확인하고 수동으로 판단합니다."

"자동화가 처리한 결과와
사람의 최종 검토가 함께 작동하는 구조입니다."

---

"정리하면,"

"차량 감지부터 속도 계산, 위반 판정, 번호판 인식, 저장까지
모든 과정이 하나의 자동화된 파이프라인으로 연결되어 있습니다."

"이 과정이 실시간으로 반복되면서
지속적으로 단속이 이루어집니다."

"즉, 사람이 직접 개입하지 않아도
실시간으로 단속이 이루어지는 구조를 구현했습니다."

갑자기 고장나면"

"저희는 이를 사전에 방지하기 위해
AI 기반 예지보전 기능을 설계했습니다."

"두 번째는 고장 유형 분류 모델입니다."

"어떤 종류의 고장이 발생할 가능성이 높은지를
예측하는 모델입니다."

"이를 통해 단순히 위험 여부뿐만 아니라,"

"어떤 부위를 점검해야 하는지까지
사전에 판단할 수 있도록 구현할 예정입니다."

'사후 대응'이 아닌 '사전 예방' 중심의 관제 시스템으로 확장할 계획입니다."

5. DB 설계 ERD
======================================================================

"이번에는 저희 시스템의 데이터 구조와 DB 설계를 설명드리겠습니다."

"저희는 실시간 교통 관제 시스템의 특성을 고려하여,
데이터 흐름과 확장성을 중심으로 DB를 설계했습니다.
특히 AI 서버, ITS 데이터, 관제 시스템이 유기적으로 연결되도록 구성했습니다."

"데이터베이스는 PostgreSQL 17을 사용하였으며,
현재 시스템은 실제 운영에 사용되는 기준으로
총 11개의 테이블로 구성되어 있습니다."

"테이블은 기능별로 네 가지 영역으로 나누었습니다."

"인증, 인프라 관리, 단속 및 예지보전,
그리고 교통 데이터 영역입니다."

"이처럼 기능별로 분리한 이유는
각 영역의 책임을 명확히 하고,
한 영역의 변경이 다른 영역에 영향을 주지 않도록 하기 위해서입니다."

---

"먼저 인증 영역입니다."

"member 테이블 하나로 구성되어 있으며,
관제 담당자의 로그인 정보를 관리합니다."

"비밀번호는 BCrypt로 암호화하여 저장하여
보안을 강화했습니다."

"이 테이블은 시스템의 모든 인증 흐름에 사용됩니다."

---

"다음은 인프라 관리 영역입니다."

"intersection과 equipment 테이블을 중심으로 구성됩니다."

"이 영역의 핵심은
intersection을 기준으로 설비가 연결되는 구조입니다."

(ERD 가리키기)

"intersection은 교차로의 위치와 신호 정보를 관리하는 중심 테이블입니다."

"대시보드 지도에 표시되는 교차로 마커와
신호 주기 정보가 모두 이 테이블에서 조회됩니다."

"equipment는 CCTV, 신호등 등
각 설비 정보를 관리하며
intersection을 참조하는 구조입니다."

"한편 cctv_info는
ITS에서 수집한 CCTV 스트림 정보를 관리하는 테이블로,"

"내부 교차로 구조와 분리된 외부 데이터로,
독립적으로 관리하고 있습니다."

"CCTV 모니터링 화면의 12개 채널 스트림 URL이
이 테이블에서 조회됩니다."

---

"세 번째는 단속 및 예지보전 영역입니다."

"이 영역은 AI 분석 결과와 직접 연결되는 핵심 구조입니다."

"violation_log는
AI가 감지한 위반 정보를 저장하는 테이블입니다."

"번호판, 속도, 단속 이미지, 처리 상태 등을 포함하며,"

"OCR 신뢰도에 따른 검토 필요 여부를 컬럼으로 관리하여
자동 처리와 수동 검토 흐름을 분리했습니다."

"FastAPI AI 서버에서 발생한 이벤트가
Webhook을 통해 이 테이블로 직접 저장되어
실시간으로 반영됩니다."

"단속 화면의 테이블과 상세 패널이
모두 이 테이블을 실시간으로 조회합니다."

"sensor_log는 설비의 센서 데이터를 저장하고,"

"maintenance_log는 설비 유지보수 이력을 관리합니다."

"두 테이블 모두 equipment를 기준으로 연결되며,
예지보전 화면의 위험도 분석에 활용됩니다."

---

"마지막은 교통 데이터 영역입니다."

"its_route와 its_collect_log는
ITS API 기반 데이터 수집 이력을 관리합니다."

"traffic_flow_log는 도로 속도와 혼잡도를,
weather_log는 기상 정보를 저장합니다."

"이 데이터들은 대시보드의 교통량 위젯과
날씨 위젯에 직접 활용됩니다."

---

"전체 구조를 보면,"

"intersection을 중심으로 설비와 교통 데이터가 연결되고,"

"AI 서버에서 생성된 단속 데이터까지
하나의 흐름으로 통합되어 있습니다."

"즉, 데이터 수집부터 분석, 저장, 시각화까지
하나의 흐름으로 이어지는 통합 구조를 설계했습니다."

"이 구조를 통해 각 기능이 분리되어 있으면서도
데이터 흐름은 끊기지 않도록 설계했습니다."

"이 문제는 크게 두 가지로 나뉩니다."

"첫째, 저해상도 CCTV 환경의 한계입니다."

"이 문제의 핵심은
번호판 크롭 이미지의 해상도가 절대적으로 부족하다는 점입니다."

"저희가 사용하는 강화도 ITS CCTV는
공공 인프라 특성상 해상도에 제한이 있습니다."

"YOLO가 차량을 감지하고 번호판 영역을 크롭하면,
그 이미지가 수십 픽셀 수준으로 줄어들고,
그 상태에서는 OCR이 문자를 구분하지 못합니다."

"이를 개선하기 위해
밝기 보정, 이진화, 컨투어 탐지, 슬라이딩 윈도우 등
여러 방법을 단계적으로 적용했습니다."

"그러나 결론적으로,
입력 데이터 해상도가 부족한 경우
알고리즘만으로는 해결할 수 없는,
데이터 자체의 한계라는 점을 확인했습니다."

"따라서 현재는 기능 검증을 위해
샘플 번호판 이미지로 OCR 파이프라인을 구성한 상태입니다."

---

"둘째, OCR 결과 검증 로직의 버그입니다."

"번호판 인식 결과를 검증하는 정규식에서
'년', '월'과 같은 비정상 문자열이
번호판으로 통과되는 문제가 있었습니다."

"정규식을 실제 번호판 규격에 맞게 수정하여
현재는 해결된 상태입니다."

---

"두 번째는 예지보전 모델의 정확도 문제입니다."

"현재 모델 정확도는 약 60% 수준으로,
목표로 하는 80% 이상에 미치지 못하고 있습니다."

"예지보전 모델은 설비 고장을 사전에 예측하는 모델인데,
잘 학습시키려면 실제 고장이 난 데이터가 많이 필요합니다."

"그런데 현실에서 교통 설비는 쉽게 고장나지 않습니다."

"즉, 고장 데이터 자체가 부족한 것이
정확도 한계의 가장 큰 원인입니다."

"이를 극복하기 위해
온도, 진동, 전류 등 센서 데이터 기반 피처 엔지니어링과
합성 데이터 생성을 병행하고 있으며,
XGBoost 모델의 하이퍼파라미터 튜닝을 지속적으로 진행 중입니다."

---

"두 이슈 모두 공통적으로
현실 데이터의 한계에서 발생한 문제입니다."

"저희는 이 한계를 단순히 우회하는 것이 아니라,
실제 환경에서 적용 가능한 수준까지 끌어올리는 것을 목표로
개선하고 있습니다."

"즉, 이 문제는 기술의 문제가 아니라
데이터 환경의 문제라는 점을 명확히 확인할 수 있었습니다."

7. 향후 계획
======================================================================

"MVP 구현을 마무리한 시점에서,
향후 계획은 핵심 기능 고도화와 추가 기능 확장입니다."

---

"먼저 핵심 기능 개선입니다."

"번호판 인식의 경우,
현재 CCTV 해상도 한계로 인해
실시간 인식에 어려움이 있습니다."

"YOLO 기반 번호판 전용 탐지 모델로 영역을 먼저 검출하고,
해당 이미지만 OCR에 전달하는 2단계 구조로 개선할 계획입니다."

"즉, 이 기능은 시스템 신뢰도를 결정하는 핵심 요소입니다."

---

"설비 예지보전은
현재 모델 정확도가 목표 수준에 미치지 못하고 있습니다."

"데이터 보강과 하이퍼파라미터 튜닝을 통해
성능을 지속적으로 개선할 예정입니다."

"단순 참고 수준이 아니라,
실제 운영에 사용할 수 있는 수준까지 끌어올리는 것이 목표입니다."

---

"다음은 추가 기능 확장입니다."

"시간이 허용된다면 세 가지 기능을 우선적으로 고려하고 있습니다."

---

"첫 번째는 실선 침범 단속입니다."

"현재 CCTV에서 차선 식별이 가능하며,
백엔드에도 위반 타입이 이미 정의되어 있어"

"AI 감지 로직만 추가하면
바로 확장 가능한 상태입니다."

---

"두 번째는 알림 서비스입니다."

"위반 발생이나 설비 이상 징후를
관리자에게 실시간으로 전달하는 기능으로,"

"SSE 또는 WebSocket 기반으로 구현을 고려하고 있습니다."

---

"세 번째는 교통 통계 리포트입니다."

"수집된 데이터를 시간대별, 구간별로 시각화하여"

"단순 조회를 넘어
교통 흐름 분석까지 가능하도록 확장할 계획입니다."

---

"이러한 기능들은 모두 현재 아키텍처 위에서
자연스럽게 확장 가능한 구조로 설계되어 있습니다."

"남은 기간 동안 우선순위를 정해
단계적으로 구현해 나갈 계획입니다."

"단순한 단속 시스템을 넘어,
데이터 기반 교통 관제 플랫폼으로 발전시키는 것이 목표입니다."

"즉, 사람이 보는 시스템이 아니라
스스로 판단하는 시스템으로 발전시키는 것이 목표입니다."
