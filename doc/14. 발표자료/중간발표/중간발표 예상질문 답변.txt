1. 호모그래피 / 속도 계산
======================================================================

Q1. 기준점 4개를 실제 도로에서 어떻게 측정했나요?

A. 현장 직접 측정은 불가능했기 때문에, 카카오맵·네이버지도 위성 이미지의 거리 측정
   기능을 활용했습니다. 강화대교 인근 4차선 도로에서 차선 폭, 정지선 간격 등 실측
   가능한 기준점을 선정하고, 픽셀 좌표와 위성 기반 실세계 미터 좌표를 매핑하여
   호모그래피 행렬을 계산했습니다. 정밀 측량 장비를 쓴 것은 아니기 때문에 오차가
   있을 수 있으며, 실제 운영 환경에서는 현장 실측으로 보정이 필요하다는 점을
   인지하고 있습니다.

----------------------------------------------------------------------

Q2. EMA 알파값 0.3으로 정한 근거는 무엇인가요?

A. 알파값이 높을수록 최근 프레임에 가중치가 집중되어 반응성은 좋아지지만 노이즈에
   취약해지고, 낮을수록 스무딩이 강해지지만 실제 속도 변화에 느리게 반응합니다.
   0.1, 0.2, 0.3, 0.5를 각각 적용해 같은 차량 구간에서의 속도 그래프를 비교했으며,
   0.3이 급격한 튀는 값은 제거하면서도 실제 가속·감속 패턴을 적절히 반영하는
   균형점이라고 판단했습니다.

----------------------------------------------------------------------

Q3. 70km/h 초과를 5프레임 연속으로 판정하는 기준 근거는?

A. 단일 프레임 판정 시 카메라 노이즈나 일시적 좌표 오차로 인한 오탐이 빈번하게
   발생했습니다. 프레임 수를 늘릴수록 오탐은 줄지만 실제 과속 차량을 놓칠 가능성이
   높아집니다. 30FPS 기준으로 5프레임은 약 0.17초에 해당하며, 이 구간을 지속적으로
   초과한 경우를 실질적 과속으로 판단하기에 적합한 최소 단위라고 실험적으로 결론
   내렸습니다. 이 값 또한 실운영 환경에서는 조정이 필요할 수 있습니다.

2. OCR / 번호판 인식
======================================================================

Q4. 샘플 이미지를 쓴다면 지금 데모가 실제 단속이라고 볼 수 있나요?

A. 정확히 말씀드리면, 현재 데모는 실 단속이 아니라 end-to-end 파이프라인 검증입니다.
   실제 CCTV에서 과속 차량을 감지하고 위반 이벤트를 생성하는 부분까지는 실시간으로
   동작합니다. 다만 그 시점에 CCTV 원본 번호판 이미지가 해상도 부족으로 OCR 인식이
   불가하기 때문에, 해당 단계에서만 샘플 번호판 이미지로 대체하여 이후 OCR → 백엔드
   저장 → 화면 표시 흐름을 검증하고 있습니다. 즉 감지·속도 계산·저장·시각화
   파이프라인은 실동작이고, 번호판 인식 정확도가 현재 시스템의 남은 과제입니다.

----------------------------------------------------------------------

Q5. OCR 신뢰도 0.85 기준은 어떻게 설정했나요?

A. 샘플 번호판 이미지에 대해 PaddleOCR이 반환하는 신뢰도 분포를 분석했습니다.
   정상 인식 케이스는 대부분 0.85 이상에 분포하고, 오인식 케이스는 그 아래에 집중되는
   경향이 있었습니다. 0.85를 기준으로 했을 때 자동 처리 비율과 수동 검토 비율의 균형이
   가장 합리적이었고, 이 기준 미만은 이미지를 직접 확인해야 하는 케이스로 분류하여
   사람이 최종 판단하도록 설계했습니다.

----------------------------------------------------------------------

Q6. YOLO 번호판 전용 모델 학습에 필요한 데이터셋은 어떻게 확보할 계획인가요?

A. AI Hub에서 제공하는 한국 번호판 데이터셋을 1차로 활용할 계획입니다. 추가로 현재
   강화도 CCTV에서 수집되는 차량 영상에서 번호판 영역을 직접 라벨링하여 도메인 특화
   데이터를 보강할 예정입니다. 다만 실제 번호판 이미지는 개인정보에 해당하기 때문에,
   학습용 데이터 처리 과정에서 비식별화 절차를 거치는 방향으로 검토하고 있습니다.

3. 예지보전
======================================================================

Q7. 정확도 60%면 실제 운영 시 어떤 문제가 생기나요? 오탐과 미탐 중 어느 쪽이 더 위험한가요?

A. 예지보전 맥락에서는 미탐(고장 예측 실패)이 더 위험합니다. 오탐은 불필요한 점검
   비용을 유발하지만, 미탐은 실제 고장으로 이어져 관제 공백이 생기기 때문입니다.
   현재 60% 수준은 참고 지표 역할로는 활용 가능하지만, 독립적인 의사결정 기준으로
   사용하기에는 부족하다고 판단하고 있습니다. 따라서 현재는 센서 데이터 이상 탐지와
   병행 운영하여 상호 보완하는 구조를 목표로 개선 중입니다.

----------------------------------------------------------------------

Q8. 합성 데이터로 학습한 모델이 실제 설비 예측에 유효한가요?

A. 합성 데이터의 한계는 인식하고 있습니다. 합성 데이터는 실제 물리적 고장 패턴의
   복잡성을 완전히 재현하기 어렵기 때문에, 모델이 합성 패턴에만 과적합될 위험이 있습니다.
   이를 보완하기 위해 합성 데이터 비율을 제한하고 실제 센서 로그 기반 데이터를 최대한
   확보하는 방향으로 진행하고 있습니다. 최종적으로는 실제 고장 이력 데이터가 확보될수록
   모델 신뢰도가 올라가는 구조입니다.

4. 아키텍처 / 시스템 안정성
======================================================================

Q9. AI 서버 장애 시 위반 감지 공백은 어떻게 처리하나요?

A. 현재 구조에서는 AI 서버가 다운되면 해당 시간의 과속 감지는 중단됩니다. CCTV
   모니터링 화면은 백엔드 경유 스트림이 아닌 ITS 직접 스트림이기 때문에 별도로
   유지됩니다. AI 서버 장애 감지는 현재 헬스체크 엔드포인트로 모니터링할 수 있으며,
   향후 알림 서비스 구현 시 AI 서버 다운 상태를 관제자에게 즉시 알리는 기능을
   추가할 계획입니다.

----------------------------------------------------------------------

Q10. DLQ 재전송 5분 공백 동안 운영자는 어떻게 알 수 있나요?

A. 현재 단계에서는 DLQ 상태를 별도 화면에서 직접 확인하는 구조는 구현되어 있지 않습니다.
   단속 데이터 유실 방지를 우선으로 설계했고, 운영자 알림 연동은 향후 알림 서비스
   (SSE/WebSocket) 구현 시 DLQ 이벤트도 포함하여 실시간 통보하는 방향으로 확장할
   계획입니다.

----------------------------------------------------------------------

Q11. 12채널 동시 스트리밍 시 서버 리소스는 어느 정도인가요?

A. 로컬 환경(i7, 16GB RAM) 기준으로 12채널 동시 재생 시 CPU 사용률은 약 30~40%
   수준입니다. MJPEG 방식은 H.264 같은 코덱 대비 압축률이 낮지만, 서버 사이드
   디코딩 부하가 없고 구현이 단순하다는 장점이 있습니다. 각 채널의 해상도와 FPS를
   제한하여 안정적인 동시 재생이 가능하도록 조정했으며, 채널 수가 늘어날 경우에는
   스트리밍 서버를 별도 분리하는 방향으로 확장 가능합니다.

Q12. ITS CCTV 영상 접근 권한은 어떻게 확보했나요?

A. 국토교통부 ITS 국가교통정보센터에서 제공하는 공개 CCTV 스트림 URL을 사용했습니다.
   별도 신청 절차 없이 공개된 스트림에 접근하는 방식이며, 상업적 목적이 아닌
   학습·연구 목적으로 활용하고 있습니다.

6. 확장성 / 실용성
======================================================================

Q13. 다른 지역으로 확장 시 추가 작업이 얼마나 필요한가요?

A. 교차로마다 호모그래피 기준점을 새로 설정해야 하기 때문에, 신규 지역 추가 시
   해당 교차로의 픽셀-실세계 좌표 매핑 작업이 필요합니다. DB의 intersection 테이블과
   cctv_info 테이블에 새 교차로 정보를 등록하면 나머지 파이프라인은 동일하게 동작합니다.
   즉 핵심 알고리즘 수정 없이 데이터 설정만으로 확장이 가능한 구조로 설계했습니다.

----------------------------------------------------------------------

Q14. 실제 행정 단속에 사용되려면 법적 요건을 검토해 보셨나요?

A. 도로교통법상 무인 단속 장비는 경찰청 또는 지자체의 공식 인증과 형식 승인을 받아야
   합니다. 저희 시스템은 현재 기술적 가능성을 검증하는 수준으로, 실제 법적 단속 효력을
   갖추려면 장비 인증, 개인정보 처리방침 수립, 이의신청 절차 등 제도적 요건이 추가로
   필요합니다. 이 프로젝트는 기술 구현 관점에서의 MVP이며, 실 운영 전환 시에는 해당
   요건을 갖춰야 한다는 점을 전제로 개발하고 있습니다.

7. 추가 날카로운 질문

Q15. YOLO nano 정확도가 실제 야간 영상에서도 충분한가요?

A. 야간 영상에서는 차량 윤곽이 불명확해 감지율이 낮아지는 것은 사실입니다. 실험 결과
   주간 대비 야간에서 감지 신뢰도가 약 10~15% 낮게 나오는 경향이 있었습니다. 이를
   보완하기 위해 신뢰도 임계값을 야간 시간대에 소폭 낮추는 방식을 검토하고 있습니다.
   근본적인 해결은 야간 데이터를 포함한 파인튜닝이 필요하며, 이 또한 향후 개선 과제로
   인식하고 있습니다.

Q16. 과속 판정 기준 70km/h는 DB에서 관리하나요, 하드코딩인가요?

A. 현재는 FastAPI 서버의 설정값으로 관리되고 있습니다. intersection 테이블에 제한 속도
   컬럼이 정의되어 있어, 향후 교차로별로 다른 제한 속도를 DB에서 읽어 동적으로 적용하는
   구조로 전환할 계획입니다. 이는 다른 구간에 확장할 때 코드 수정 없이 운영 가능하게
   만드는 작업이기도 합니다.


Q1. 왜 AI 서버를 분리했나요?

👉 답변

AI 영상 처리와 웹 서비스의 성격이 다르기 때문에 분리했습니다.

AI는 GPU 연산과 실시간 처리가 필요하고,
웹 서버는 사용자 요청 처리 중심이기 때문에
하나로 구성하면 성능 간섭이 발생할 수 있습니다.

따라서 안정성과 확장성을 고려하여
FastAPI 기반 AI 서버를 별도로 구성했습니다.
Q2. 실시간 처리라고 했는데 지연은 어느 정도인가요?

👉 답변

현재 기준으로 약 1~2초 내외 지연이 발생합니다.

YOLO 추론, 트래킹, 속도 계산, OCR까지 포함된 파이프라인이기 때문에
완전한 실시간보다는 준실시간 수준입니다.

향후에는 모델 경량화와 파이프라인 최적화를 통해
지연을 더 줄일 계획입니다.
🚦 2. CCTV
Q3. 왜 MJPEG를 사용했나요?

👉 답변

지연이 낮고 구현이 단순하기 때문입니다.

WebRTC는 지연은 더 낮지만 구현 복잡도가 높고,
HLS는 안정적이지만 지연이 큰 단점이 있습니다.

현재 시스템에서는 빠른 확인이 중요하기 때문에
MJPEG 방식을 선택했습니다.
Q4. 채널이 늘어나면 어떻게 하나요?

👉 답변

현재 구조는 채널 확장을 고려하여 설계되어 있습니다.

스트림 단위로 처리되기 때문에
서버 자원을 분산하거나
해상도와 FPS를 조정하는 방식으로 확장이 가능합니다.
🚗 3. 단속 파이프라인
Q5. YOLO를 선택한 이유는?

👉 답변

실시간 처리 성능 때문입니다.

다른 모델 대비 속도가 빠르고,
단일 네트워크로 탐지와 분류를 동시에 수행할 수 있어
영상 처리에 적합하다고 판단했습니다.
Q6. nano 모델 쓰면 정확도 떨어지지 않나요?

👉 답변

맞습니다, 일부 정확도 손해는 있습니다.

하지만 관제 시스템에서는
속도가 더 중요한 요소라고 판단했고,
실시간 처리 안정성을 우선으로 선택했습니다.

향후에는 모델 경량화와 함께 정확도 개선도 고려하고 있습니다.
Q7. ByteTrack 선택 이유는?

👉 답변

낮은 confidence 객체까지 활용할 수 있기 때문입니다.

CCTV 환경에서는 차량이 가려지거나
탐지 신뢰도가 순간적으로 떨어지는 경우가 많습니다.

ByteTrack은 이런 상황에서도
추적을 유지할 수 있어 적합하다고 판단했습니다.
Q8. 호모그래피 정확도는 어떻게 검증했나요?

👉 답변

실제 도로 기준 거리와 비교하여 검증했습니다.

기준 구간을 설정한 후
계산된 이동 거리와 실제 거리를 비교하여
오차 범위를 확인하는 방식으로 검증했습니다.

완벽한 정확도는 아니지만
속도 계산에 활용 가능한 수준으로 판단했습니다.
Q9. 카메라 각도 바뀌면?

👉 답변

호모그래피는 카메라 기준이 고정되어야 합니다.

각도가 바뀌면 행렬을 다시 계산해야 하는 한계가 있습니다.

따라서 현재는 고정 CCTV 기준으로 설계되어 있으며,
변경 시 재보정이 필요합니다.
Q10. 왜 5프레임 기준인가요?

👉 답변

오탐 방지를 위한 기준입니다.

단일 프레임 기준으로는 노이즈 영향이 크기 때문에,
일정 시간 이상 유지되는 경우만 위반으로 판단하도록 설계했습니다.

여러 실험을 통해 안정적인 기준으로 5프레임을 선택했습니다.
🔤 4. OCR
Q11. OCR 정확도는 어느 정도인가요?

👉 답변

샘플 이미지 기준에서는 높은 정확도를 보이지만,
실제 CCTV 환경에서는 해상도 문제로 정확도가 낮아집니다.

이 부분은 현재 시스템의 가장 큰 한계 중 하나입니다.
Q12. OCR 문제 해결 못 한 거 아닌가요?

👉 답변 (🔥 중요)

맞습니다, 완전히 해결하지는 못했습니다.

다만 문제의 원인을 데이터 해상도 한계로 명확히 규명했고,
전처리와 파이프라인 개선을 통해
최대한 성능을 끌어올렸습니다.

현재는 기능 검증 단계로 보고 있으며,
향후 개선을 진행할 계획입니다.
🗄️ 5. DB
Q13. PostgreSQL 선택 이유는?

👉 답변

안정성과 확장성 때문입니다.

관계형 데이터 구조에 적합하고,
트랜잭션 처리와 확장성이 뛰어나
실시간 데이터 처리에 적합하다고 판단했습니다.
Q14. intersection 중심 구조 장점은?

👉 답변

데이터 구조를 단순화할 수 있다는 점입니다.

교차로를 기준으로 설비와 데이터를 연결하면
조회와 확장이 직관적으로 이루어집니다.

새로운 교차로 추가도 쉽게 확장 가능합니다.
🛠️ 6. 개발 이슈
Q15. 정확도 60%면 쓸 수 있나요?

👉 답변 (🔥 중요)

현재 단계에서는 참고용 수준입니다.

실제 운영에 사용하기에는 부족하지만,
모델 구조와 파이프라인 검증은 완료된 상태입니다.

향후 데이터 확보를 통해
실사용 수준으로 개선하는 것이 목표입니다.
🚀 7. 향후 계획
Q16. 실제 구현 가능한 계획인가요?

👉 답변

현재 아키텍처를 기반으로 충분히 구현 가능하다고 판단했습니다.

기존 구조 위에서 확장하는 형태이기 때문에
전체 구조를 변경하지 않고도 단계적으로 적용 가능합니다.
💥 8. 압박 질문
Q17. 이 시스템 실제로 쓸 수 있나요?

👉 답변 (🔥 핵심)

현재는 완전한 상용 수준은 아니지만,
핵심 기능은 실제 동작 가능한 수준까지 구현했습니다.

특히 데이터 흐름과 자동화 파이프라인은
실제 시스템에 적용 가능한 구조라고 생각합니다.
