[Slide 7] 해결 전략: MSA 독립 서버 구조

그렇다면 이 세 가지 문제를 어떻게 해결했을까요.

핵심 전략은 역할별 독립 서버 구조, 마이크로서비스 아키텍처의 도입입니다.

AI-PASS는 세 개의 서버가 역할을 분담하고 실시간으로 연동됩니다.

CCTV 영상이 AI 서버로 들어오면, AI가 차량을 추적하고 위반 여부를 판단합니다.

고비용 AI 연산을 분리했기 때문에 웹 서비스 응답 속도 저하를 원천 차단했고,
기능 확장 시 해당 서버만 독립적으로 수정할 수 있습니다.

[Slide 9] 기술 스택: Frontend & Backend
======================================================================

각 서버의 기술 선택 이유를 말씀드리겠습니다.

화면 서버는 Vue 3와 Vite를 채택했습니다.
CCTV 격자 화면, 대시보드 등 독립적인 화면 구성 요소가 많은 관제 화면의 특성상
컴포넌트 단위 관리가 용이한 Vue 3가 적합했습니다.

백엔드는 Spring Boot와 MyBatis, PostgreSQL입니다.

인증은 서버 세션 기반 요청 차단 구조를 구현했습니다.
AI-PASS는 제한된 관제 담당자 환경이므로
복잡한 토큰 관리보다 HttpSession과 커스텀 인터셉터 기반의
단순하고 안정적인 접근 제어가 더 적합하다고 판단했습니다.
Spring Security는 사용하지 않았습니다.

데이터베이스 접근은 MyBatis XML 방식을 사용했습니다.
교통 데이터는 다중 조건 필터링과 복잡한 집계 쿼리가 많기 때문에
SQL을 직접 제어할 수 있는 구조가 필요했습니다.

======================================================================
[Slide 10] 기술 스택: AI Vision & Data Infrastructure
======================================================================

AI 서버는 FastAPI를 기반으로 구축했습니다.
YOLO, OpenCV, PaddleOCR을 제약 없이 통합할 수 있고,
비동기 처리로 영상 전송과 실시간 웹훅 전송을 동시에 처리합니다.

차량 감지에는 YOLO Nano 모델을 적용했습니다.
12채널 실시간 영상을 동시에 처리해야 하므로
무거운 대형 모델 대신 가볍고 빠른 처리량의 모델을 선택했습니다.

차량 추적은 ByteTrack 방식으로, 야간이나 부분 가림 등
악조건에서도 동일 객체 추적을 유지합니다.

번호판 인식은 PaddleOCR을 활용하며,
저해상도 환경을 극복하기 위한 4단계 재시도 구조를 적용했습니다.

예지보전 모델은 XGBoost 기반으로,
설비 잔여 수명 예측과 고장 유형 분류를 담당합니다.

[Slide 11] DB 설계: 14개 테이블 허브 구조
======================================================================

데이터베이스는 총 14개 테이블로 구성되며 네 가지 영역으로 분류했습니다.

인증 영역은 관제 담당자 접근 권한을 관리하는 회원 테이블입니다.

인프라 관리 영역은 교차로·설비·CCTV 세 테이블로 구성됩니다.
핵심은 교차로 테이블로, 대시보드·교통 데이터·설비 정보가
모두 이 테이블을 허브로 연결됩니다.

단속·예지보전 영역에서 단속 기록 테이블은
AI 서버의 위반 감지 결과를 웹훅으로 실시간 적재받습니다.

교통 데이터 영역은 ITS 연계로 수집한 교통량·혼잡도 데이터를
시계열로 저장하며 대시보드 위젯이 주기적으로 조회합니다.

특히 초기 설계 단계부터 위반 유형(Type) 항목을 포함해두었기 때문에,
실선 침범 기능 추가 시 기존 구조 수정 없이 AI 서버 로직만 추가해 즉시 확장할 수 있었습니다.

[Slide 13] 시연: 12채널 실시간 CCTV 모니터링

강화도 국도 48호선 12개 채널을 한 화면에서 동시에 실시간으로 확인합니다.

물리적으로 분리한 이중 처리 구조를 설계했습니다.
덕분에 AI 연산 부하와 무관하게 24시간 끊김 없는 관제 환경이 유지됩니다.

[Slide 14] 시연: 과속 단속 파이프라인 (1) — 감지·추적·속도산출
======================================================================

네 번째는 AI 기반 과속 단속입니다.

전체 흐름은 세 단계로 먼저 설명드리겠습니다.

1단계 — 차량 감지.
YOLO Nano 모델이 실시간 영상에서 차량을 즉각 탐지합니다.

2단계 — 객체 추적.
ByteTrack이 각 차량에 고유 ID를 부여하고 프레임 간 궤적을 추적합니다.
여러 프레임에 걸쳐 같은 차량을 동일 개체로 인식해야 속도를 계산할 수 있기 때문입니다.

3단계 — 속도 산출.
CCTV는 2차원 영상이기 때문에 픽셀 이동만으로는 실제 거리를 알 수 없습니다.
이를 해결하기 위해 호모그래피 변환을 적용했습니다.
강화대교 실제 도로 위 기준점 4개를 직접 실측하여
픽셀 좌표를 실제 미터 단위 거리로 매핑했습니다.
여기에 지수이동평균을 적용해 노이즈로 인한 속도 불안정을 안정화했습니다.

(시연 진행)

======================================================================
[Slide 15] 시연: 과속 단속 파이프라인 (2) — OCR 4단계 & 웹훅 전송
======================================================================

위반이 확정되면 번호판 인식 단계로 넘어갑니다.

저해상도 CCTV 환경에서 인식률을 끌어올리기 위해
4단계 재시도 구조를 자체 설계했습니다.

1단계 — 원본 이미지 직접 인식.
2단계 — 명암 적응 보정으로 야간·역광 환경 대응.
3단계 — 이진화 대비 강화로 문자 윤곽 분리.
4단계 — 색상 분석 기반 윤곽 탐지로 번호판 영역 최종 추출.

각 단계마다 신뢰도 60% 이상, 한국 번호판 형식 통과 결과만 채택합니다.

위반이 최종 확정되면 AI 서버가 백엔드로 웹훅을 즉시 전송합니다.
단속 이미지, 번호판, 속도, 위치 정보가 하나의 패키지로 전달되어
단속 내역 화면에 실시간으로 기록됩니다.

(시연 진행)

======================================================================
[Slide 16] 시연: 실선 침범 단속
======================================================================

다섯 번째는 실선 침범 단속입니다.

앞서 보신 과속 감지와 완전히 동일한 파이프라인을 사용합니다.
YOLO Nano로 차량을 감지하고 ByteTrack으로 추적하되,
속도 산출 대신 ROI 기반 차선 침범 판정 로직을 탑재했습니다.

도로 위에 설정된 가상 관심 영역을
차량의 이동 궤적이 통과하는 순간을 위반으로 판독합니다.

이 기능을 즉시 추가할 수 있었던 이유가 바로 초기 설계의 확장성 덕분입니다.
기존 백엔드나 DB 구조 수정 없이 AI 서버의 감지 로직만 추가했고,
DB 단속 기록 테이블에 이미 위반 유형 항목이 있었기 때문입니다.
불법 유턴, 역주행 등으로도 동일한 방식으로 즉시 확장 가능합니다.

(시연 진행)

[Slide 17] 시연: 설비 예지보전
======================================================================

여섯 번째는 설비 예지보전입니다.

CCTV나 신호기가 멈추는 순간 관제 공백이 발생합니다.
저희는 이 문제를 고장 전에 예측하고 선제 대응하는 구조로 해결했습니다.

온도·진동·전류 세 가지 센서 데이터를 수집하고,
XGBoost 모델이 이를 분석해 설비의 잔여 수명을 예측합니다.

예측 결과는 위험·경고·주의·정상 4단계 등급으로 표시되어
담당자가 점검 우선순위를 한눈에 파악할 수 있습니다.

모델 학습 과정에서 실제 고장 데이터가 극히 희소한 문제가 있었습니다.
이를 합성 데이터 생성 기법으로 해결했으며,
정상 데이터에 물리적 고장 특성을 반영한 패턴을 합성하여 학습 데이터를 확보했습니다.

(시연 진행)
