## 🔴 **CRITICAL #3 (추가): CctvDTO `updatedAt` 타입 불일치**

### 문제 설명
CctvDTO의 `updatedAt` 필드가 `String`으로 정의되어 있는데, 이는 DB 스키마와 일관성이 없습니다.

### 현황 비교

**CctvDTO.java (Line 9)**
```java
private String updatedAt;  // ❌ String 타입
```

**cctv.xml (Line 17, 33, 50)**
```xml
TO_CHAR(updated_at AT TIME ZONE 'Asia/Seoul', 'YY-MM-DD HH24:MI:SS') AS updated_at
-- ✅ SQL에서 명시적으로 TO_CHAR로 문자열 변환
```

**DB 스키마 (cctv_info 테이블)**
```sql
updated_at TIMESTAMP  -- 실제는 TIMESTAMP 타입
```

### 불일치의 의미
- ✅ **현재 상태**: SQL에서 TIMESTAMP를 문자열로 변환 → DTO에서 String 수신
- ⚠️ **문제점**: 
  1. JSON 응답 시 타입이 일관되지 않음 (ISO 8601 vs 포맷된 문자열)
  2. 프론트엔드에서 날짜 포매팅 불가능 (이미 포맷되어 있으므로)
  3. 향후 유지보수 시 헷갈림

### 권장 해결 방법

**Option 1 (권장): LocalDateTime으로 통일**
```java
// CctvDTO.java
import java.time.LocalDateTime;

private LocalDateTime updatedAt;  // ✅ LocalDateTime 타입

public LocalDateTime getUpdatedAt() { return updatedAt; }
public void setUpdatedAt(LocalDateTime updatedAt) { this.updatedAt = updatedAt; }
```

```xml
<!-- cctv.xml -->
<select id="findAllActive" resultType="com.aipass.dto.CctvDTO">
    SELECT
        cctv_id,
        ...
        updated_at  -- ✅ TIMESTAMP → LocalDateTime 자동 변환 (MyBatis)
    FROM cctv_info
    WHERE is_active = true
</select>
```

**Option 2: String 유지 (현재 방식 문서화)**
```java
// CctvDTO.java
/**
 * CCTV 마지막 업데이트 시간 (포맷: YY-MM-DD HH24:MI:SS, Asia/Seoul)
 * JSON 응답 시 직렬화되며, 추가 포매팅이 필요 없습니다.
 */
private String updatedAt;
```

```xml
<!-- cctv.xml - 명시적 주석 추가 -->
<select id="findAllActive" resultType="com.aipass.dto.CctvDTO">
    SELECT
        cctv_id,
        ...
        TO_CHAR(updated_at AT TIME ZONE 'Asia/Seoul', 'YY-MM-DD HH24:MI:SS') AS updated_at
        <!-- 주의: String으로 반환되므로 프론트엔드에서 재포매팅 불필요 -->
    FROM cctv_info
</select>
```

## 🟠 **HIGH #4 (추가 발견): CctvMapper의 하드코딩된 cctvId 문제**

### 문제 설명
`cctv.xml`의 `findAiTargetCctv()` 메서드가 특정 cctvId(`'4700'`)로 하드코딩되어 있습니다.

### 현황 분석

**cctv.xml (Line 35)**
```xml
<select id="findAiTargetCctv" resultType="com.aipass.dto.CctvDTO">
    SELECT ...
    FROM cctv_info
    WHERE cctv_id = '4700'  <!-- ❌ 하드코딩: cctvId='4700' 고정 -->
      AND is_active = true
    LIMIT 1
</select>
```

**CctvController.java (Line 44-62)**
```java
@GetMapping("/ai-target")
public ResponseEntity<?> getAiTargetCctv() {
    try {
        CctvDTO data = cctvMapper.findAiTargetCctv();  // ❌ 파라미터 없음
        // ...
    }
}
```

### 영향받는 부분

1. **FastAPI (배영환 담당)**
   - `/api/cctv/ai-target` 엔드포인트 호출 시 항상 cctvId='4700'만 반환
   - 다른 CCTV로 처리 변경 불가능
   - 복수 CCTV 순회 처리 불가능

2. **설정 변경 불가**
   - DB에 cctvId='4700'이 없으면 null 반환
   - 런타임 중 대상 CCTV 변경 불가능

### 권장 해결 방법

**Option 1: 동적 파라미터화**
```xml
<!-- cctv.xml -->
<select id="findAiTargetCctv" parameterType="String" resultType="com.aipass.dto.CctvDTO">
    SELECT ...
    FROM cctv_info
    WHERE cctv_id = #{cctvId}
      AND is_active = true
    LIMIT 1
</select>
```

```java
// CctvMapper.java
CctvDTO findAiTargetCctv(String cctvId);  // 파라미터 추가
```

**Option 2: 설정 테이블**
```sql
-- ai_config 테이블 추가
CREATE TABLE ai_config (
    config_id BIGSERIAL PRIMARY KEY,
    key VARCHAR(100) NOT NULL UNIQUE,
    value VARCHAR(500),
    description TEXT,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

INSERT INTO ai_config (key, value, description) VALUES
('ai_target_cctv_id', '4700', 'AI 서버가 처리할 대상 CCTV ID');
```

```java
// 컨트롤러에서 설정값 읽어서 조회
String targetCctvId = configService.getValue("ai_target_cctv_id");
CctvDTO data = cctvMapper.findAiTargetCctv(targetCctvId);
```

**Option 3: 첫 번째 활성 CCTV 반환**
```xml
<!-- cctv.xml -->
<select id="findAiTargetCctv" resultType="com.aipass.dto.CctvDTO">
    SELECT ...
    FROM cctv_info
    WHERE is_active = true
    ORDER BY cctv_id ASC  <!-- ✅ 첫 번째를 자동으로 선택 -->
    LIMIT 1
</select>
```

## 🟠 **HIGH #5 (추가 발견): CctvDTO와 CctvInfoDTO의 역할 혼동**

### 문제 설명
거의 동일한 두 DTO가 존재하여 혼동을 초래합니다.

### 현황 비교

**CctvDTO (cctv.xml 기반)**
```java
private String cctvId;        // cctv_id
private String cctvName;      // cctv_name
private String roadName;      // road_name
private String streamUrl;     // stream_url
private String district;      // district
private String updatedAt;     // updated_at (TO_CHAR로 포매팅된 String)
private Double latitude;      // latitude
private Double longitude;     // longitude
private Boolean isActive;     // is_active
```

**CctvInfoDTO (cctv_info.xml 기반)**
```java
private String cctvId;        // cctv_id
private String cctvName;      // cctv_name
private String roadName;      // road_name
private Double latitude;      // latitude
private Double longitude;     // longitude
private String streamUrl;     // stream_url
private String district;      // district
private Boolean isActive;     // is_active
private LocalDateTime updatedAt;  // updated_at (LocalDateTime)
```

### 차이점 분석

| 필드 | CctvDTO | CctvInfoDTO |
|------|---------|-----------|
| updatedAt 타입 | String (포맷됨) | LocalDateTime |
| 사용 Mapper | CctvMapper | CctvInfoMapper |
| 사용 XML | cctv.xml | cctv_info.xml |
| 용도 | CCTV 목록 조회, 스트리밍 URL 제공 | 스케줄러에서 UPSERT |

### 영향받는 부분

1. **개발자 혼동**
   - 어느 DTO를 사용해야 하는지 불명확
   - 두 DTO 간 변환 로직 필요할 경우 복잡성 증가

2. **유지보수 어려움**
   - 필드 추가/삭제 시 두 DTO를 모두 수정해야 함
   - 불일치 위험 높음

## 🟡 **MEDIUM #7 (추가 발견): TrafficFlowHourDTO와 TrafficFlowDTO 역할 불명확**

### 문제 설명
시간별 교통량 집계(`TrafficFlowHourDTO`)와 개별 교통량 데이터(`TrafficFlowDTO`)의 역할이 명확하지 않습니다.

### 현황 분석

**TrafficFlowHourDTO.java**
```java
private String hour;             // HH24:MI 포맷
private Long vehicleCount;       // 차량 수
private Double avgSpeed;         // 평균 속도
private String congestionLevel;  // SMOOTH, SLOW, CONGESTED
```

**TrafficFlowDTO.java**
```java
private String roadName;         // 도로명
private String route;            // 노선
private BigDecimal speed;        // 속도 (단수)
private String congestion;       // 혼잡도
```

**traffic_flow_log 테이블 구조**
```sql
flow_id           BIGINT
link_id           VARCHAR
road_name         VARCHAR
speed             NUMERIC
congestion_level  VARCHAR
direction         VARCHAR
route_no          VARCHAR
collected_at      TIMESTAMP
created_at        TIMESTAMP
```

### 문제점

1. **TrafficFlowDTO의 필드가 DB와 매핑되지 않음**
   - `route` 필드는 DB에 없음 (route_no는 있음)
   - SELECT 쿼리가 이 필드들을 조회하지 않음

2. **TrafficFlowHourDTO는 시간대별 집계 데이터**
   - traffic.xml의 findFlowByHour()에만 사용
   - 다른 곳에서 사용 안 됨

3. **역할 분리 미흡**
   - 개별 데이터와 집계 데이터가 혼재됨

### 권장 해결 방법

**TrafficFlowDTO 재정의**
```java
public class TrafficFlowDTO {
    private Long flowId;              // flow_id
```
