## 상세 불일치 분석

### 🔴 **CRITICAL #1: IntersectionDTO vs DB 스키마 완전 불일치**

#### 문제 설명
`IntersectionDTO`는 DB에 존재하지 않는 필드들로 정의되어 있고, 반대로 DB의 필수 필드를 DTO에서 정의하지 않았습니다.

#### 현황 비교

**DTO 필드 (AIpassBackend/src/main/java/com/aipass/dto/IntersectionDTO.java: 4-12)**
```java
private int id;              // ✅ DB: intersection_id (Long)
private String name;         // ✅ DB: name
private String location;     // ❌ DB에 없음
private String status;       // ❌ DB에 없음 (신호 제어 상태?)
private String congestion;   // ❌ DB에 없음 (traffic_congestion 테이블과 혼동)
private int currentPhase;    // ❌ DB에 없음
private int phaseRemaining;  // ❌ DB에 없음
private int totalCycle;      // ❌ DB에 없음
```

**실제 DB 스키마 (PostgreSQL aipass.intersection)**
```sql
intersection_id BIGINT PRIMARY KEY
name            VARCHAR(255)
latitude        NUMERIC(10,8)      -- DTO에 없음
longitude       NUMERIC(10,8)      -- DTO에 없음
created_at      TIMESTAMP           -- DTO에 없음
```

#### 영향받는 코드

1. **TrafficController.java (Line 8-16, 19-27)**
   - `findAll()`: 교차로 목록 조회 → `location`, `status`, `congestion` 필드 매핑 실패
   - `findById()`: 교차로 상세 조회 → 동일 문제
   - SQL 쿼리는 `latitude`, `longitude` 만 SELECT하는데, DTO에는 `location` 필드가 필요

2. **TrafficView.vue**
   - 교차로 목록 조회 후 응답 데이터가 예상 구조와 불일치
   - 지도 핀 렌더링 시 위도/경도 정보 누락 가능성

#### 권장 해결 방법

**Option 1 (권장): DTO 전수 수정**
```java
public class IntersectionDTO {
    private Long intersectionId;  // intersection_id (타입: Long)
    private String name;
    private Double latitude;      // NUMERIC → Double
    private Double longitude;
    private LocalDateTime createdAt;
    
    // location, status, congestion, currentPhase, phaseRemaining, totalCycle 제거
}
```

**Option 2: 신호 제어 정보 분리**
- `IntersectionDTO` → 기본 정보만 (교차로 목록, 맵 핀)
- `IntersectionSignalDTO` → 신호 제어 상태 (별도 테이블/뷰)

### 🟠 **HIGH #2: ViolationDTO `detectedAt` 타입 불일치**

#### 문제 설명
DTO에서는 `detectedAt`을 `String`으로 정의하지만, 화면에서는 `LocalDateTime` 형식으로 처리 필요합니다.

#### 현황 분석

**ViolationDTO.java (Line 13)**
```java
private String detectedAt;  // ❌ String 타입
```

**violation.xml (Line 32)**
```xml
TO_CHAR(detected_at AT TIME ZONE 'Asia/Seoul', 'YY-MM-DD HH24:MI:SS') AS detected_at
```

**EnforcementView.vue**
```javascript
// 화면에서 detectedAt을 어떻게 처리하는지 확인 필요
// API 응답: { detectedAt: "26-03-25 14:30:45" } (String)
```

#### 영향받는 부분

1. **violation.xml의 SQL 변환**
   - 날짜 포맷팅: `TO_CHAR(...)`로 문자열로 변환
   - 시간대 변환: `AT TIME ZONE 'Asia/Seoul'`
   - 결과: 문자열 반환 (예: "26-03-25 14:30:45")

2. **EnforcementController (Line 98-105)**
   ```java
   @GetMapping("/violations/{id}")
   public ResponseEntity<?> getViolation(@PathVariable Long id) {
       ViolationDTO dto = violationMapper.findById(id);
       // detectedAt은 String으로 반환됨 (SQL에서 TO_CHAR 했으므로)
       return ResponseEntity.ok(Map.of("success", true, "data", dto));
   }
   ```

3. **EnforcementView.vue**
   ```javascript
   // 형식 불확실성으로 인한 화면 렌더링 오류 가능
   const { data } = await api.get('/enforcement/violations', { params })
   // data.items[].detectedAt = "26-03-25 14:30:45" 
   ```

#### 권장 해결 방법

**Option 1 (권장): 타입 명확화**
```java
// ViolationDTO.java
private LocalDateTime detectedAt;  // ✅ LocalDateTime 타입

// violation.xml
<select id="findById" parameterType="long" resultType="com.aipass.dto.ViolationDTO">
    SELECT
        violation_id,
        ...
        detected_at  -- 타입 매핑: TIMESTAMP → LocalDateTime
    FROM violation_log
    WHERE violation_id = #{id}
</select>

// MyBatis가 TIMESTAMP → LocalDateTime 자동 변환
```

**Option 2: 명시적 문자열 변환 (현재 방식 유지)**
```xml
<select id="findById" parameterType="long" resultType="map">
    SELECT
        violation_id,
        TO_CHAR(detected_at, 'YYYY-MM-DD HH24:MI:SS') AS detected_at,
        ...
    FROM violation_log
</select>

// resultMap으로 DTO에 명시적 매핑
```

### 🟡 **MEDIUM #4: TrafficController의 교차로 조회 설계 불완전**

#### 문제 설명
`TrafficController.getTrafficFlow()` 메서드가 실제로는 교차로별 교통 흐름 데이터를 조회하지 않고, 전체 24시간 교통량만 반환합니다.

#### 현황 분석

**TrafficController.java (Line 42-47)**
```java
@GetMapping("/flow/{intersectionId}")
public ResponseEntity<?> getTrafficFlow(@PathVariable Long intersectionId) {
    // ❌ intersectionId 파라미터를 전혀 사용하지 않음
    List<TrafficFlowHourDTO> flow = trafficMapper.findFlowByHour(24);
    // ❌ 전체 교통량만 조회, 교차로별 필터링 없음
    return ResponseEntity.ok(flow);
}
```

**traffic.xml (Line 35-50)**
```xml
<select id="findFlowByHour" resultType="com.aipass.dto.TrafficFlowHourDTO">
    SELECT
        TO_CHAR(DATE_TRUNC('hour', collected_at), 'HH24:MI') AS hour,
        COUNT(*) AS vehicle_count,
        ROUND(AVG(speed)::numeric, 1) AS avg_speed,
        ...
    FROM traffic_flow_log
    <!-- ❌ traffic_flow_log에 intersection_id 컬럼이 없음 -->
    <!-- ✅ link_id가 있지만, 교차로와의 연결고리 없음 -->
    WHERE collected_at >= NOW() - INTERVAL '24 hours'
    GROUP BY DATE_TRUNC('hour', collected_at)
</select>
```

**DB 스키마 검증**
```sql
-- traffic_flow_log 구조
link_id         VARCHAR  -- 도로 구간 ID
road_name       VARCHAR  -- 도로명
speed           NUMERIC  -- 평균 속도
congestion_level VARCHAR -- SMOOTH, SLOW, CONGESTED
direction       VARCHAR  -- UP, DOWN
route_no        VARCHAR  -- 노선번호
collected_at    TIMESTAMP
created_at      TIMESTAMP

-- ❌ intersection_id 컬럼 없음
-- ❌ traffic_flow_log와 intersection 간 직접 연결 불가
```

#### 영향받는 부분

1. **TrafficView.vue (Line 282)**
   ```javascript
   const res = await api.get(`/traffic/flow/${id}`)
   // selectedId에 교차로 ID를 전달하지만
   // 백엔드에서는 무시하고 전체 24시간 데이터만 반환
   ```

2. **결과**
   - 특정 교차로의 시간대별 교통량 그래프를 그릴 수 없음
   - 모든 교차로에서 동일한 전체 교통량 데이터가 표시됨

#### 근본 원인

**설계 문제:**
- `traffic_flow_log`는 ITS API에서 수집한 **도로 구간(링크) 데이터**
- `intersection`은 **교차로(교점) 데이터**
- 둘 간 매핑 테이블(link_to_intersection)이 없음
- 따라서 특정 교차로의 교통량 조회 불가능

#### 권장 해결 방법

**Option 1: Link-to-Intersection 매핑 테이블 추가**
```sql
CREATE TABLE link_to_intersection (
    mapping_id BIGSERIAL PRIMARY KEY,
    link_id VARCHAR NOT NULL,
    from_intersection_id BIGINT NOT NULL REFERENCES intersection(intersection_id),
    to_intersection_id BIGINT REFERENCES intersection(intersection_id),
    direction VARCHAR,  -- UP, DOWN
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

**Option 2: traffic_flow_log에 intersection_id 추가**
```sql
ALTER TABLE traffic_flow_log ADD COLUMN intersection_id BIGINT REFERENCES intersection(intersection_id);

-- TrafficFlowCollectScheduler에서 수집 시 intersection_id도 저장
```

**Option 3: 현재 설계 유지 (임시)**
- `/api/traffic/flow/{intersectionId}`는 폐기
- 대신 `/api/traffic/flow?direction=UP` 형식으로 변경
- 또는 전체 도로 구간의 교통량만 반환하도록 문서화

### 🟡 **MEDIUM #6: WeatherCollectScheduler의 고정 intersection_id 사용**

#### 문제 설명
날씨 수집 스케줄러가 하드코딩된 `intersection_id=1`을 항상 사용하여, 다중 지점 날씨 수집이 불가능합니다.

#### 현황 분석

**WeatherCollectScheduler.java (Line 24)**
```java
private static final long DEFAULT_INTERSECTION_ID = 1L;  // ❌ 고정값

@Scheduled(cron = "0 5 * * * *")
public void collect() {
    // ...
    WeatherLogDTO dto = new WeatherLogDTO();
    dto.setIntersectionId(DEFAULT_INTERSECTION_ID);  // ❌ 항상 1
    // ...
    weatherLogMapper.insert(dto);
}
```

**weather_log 테이블 구조**
```sql
weather_id          BIGINT PRIMARY KEY
intersection_id     BIGINT REFERENCES intersection
-- ✅ 교차로별로 날씨 저장 가능한 구조
```

#### 영향받는 부분

1. **현재 상황**
   - 강화 지역(STN=201) 날씨만 교차로 1번에 저장됨
   - 다른 교차로들의 지역 날씨 데이터 없음
   - ML 모델의 `ambient_temp` 값이 모든 교차로에서 동일함

2. **예지보전 ML 영향**
   - ExpectedTemp 계산 시 정확도 저하
   - 지역별 온도 편차 반영 불가능
   - RUL 예측 성능 저하

#### 권장 해결 방법

**Option 1: 복수 기상청 STN 수집**
```java
private static final Map<Long, String> INTERSECTION_STN = Map.ofEntries(
    Map.entry(1L, "201"),    // 강화
    Map.entry(2L, "108"),    // 서울
    Map.entry(3L, "112")     // 인천
);

@Scheduled(cron = "0 5 * * * *")
public void collect() {
    for (Map.Entry<Long, String> entry : INTERSECTION_STN.entrySet()) {
        Long intersectionId = entry.getKey();
        String stn = entry.getValue();
        
        WeatherLogDTO dto = new WeatherLogDTO();
```
